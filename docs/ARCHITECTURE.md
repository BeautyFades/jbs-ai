# JBS AI Control Tower — Architecture Evaluation & PoC Design

**Date:** 2026-07-03 · **Status:** Proof of Concept

## 1. What we're building

A centralized analytics "control tower" for a few hundred high-level users at JBS Foods:

- Claude AI chat + artifacts, connected via **MCP** to **Snowflake** and **dbt**
- Dashboards and charting
- Writeback (edit mapping tables in the UI → persist to Snowflake)
- Alerts / notifications, mobile support
- Robust auth + permissioning with **SSO** (Azure AD / Okta)
- Very modularized backend

## 2. Stack evaluation

### Option A — FastAPI + React (Vite) ✅ **Recommended**

| Concern | Assessment |
|---|---|
| AI/agent workloads | Best-in-class. Native `async` is essential: the Claude agent loop, MCP stdio/HTTP sessions, SSE/WebSocket streaming, and Snowflake queries are all I/O-bound and concurrent. The `mcp` and `anthropic` Python SDKs are async-first. |
| Modularity | FastAPI routers + dependency injection map cleanly to modules (`chat`, `dashboards`, `writeback`, `alerts`, `auth`). Each module is a router + service + repository, independently testable and extractable to its own service later. |
| Snowflake/dbt ecosystem | Python is the native language of the data stack (snowflake-connector, snowpark, dbt). One language across the API and data engineering. |
| Auth/SSO | OIDC via `authlib`/`fastapi-azure-auth` is mature. Fine-grained permissioning is explicit code (or Oso/Cerbos/OPA) rather than framework magic — better fit for row/column-level rules driven by Snowflake role hierarchies. |
| Mobile | React Native / Expo shares the API contract (OpenAPI is auto-generated by FastAPI) and much of the TypeScript client code. |
| Performance | Async workers handle hundreds of concurrent SSE streams cheaply; Django's ORM-centric sync model fights you here. |

**Cost of choosing FastAPI:** no batteries. Admin UI, migrations (use Alembic), and auth scaffolding are assembled, not included. For this product that's acceptable — the "database" is Snowflake, not Postgres-as-source-of-truth, so Django's biggest batteries (ORM + admin) mostly don't apply.

### Option B — Django + React

Strong when the app is CRUD over its *own* relational database: built-in ORM, admin, auth, permissions. Here, however:

- The system of record is **Snowflake**, accessed via its own connector — Django's ORM and admin add little.
- Async support (ASGI, `async def` views) is still second-class in the ORM and much of the ecosystem; long-lived MCP sessions and streaming agent loops are awkward.
- Django's permission model is table-level; we need policy-level (user × domain × row) authorization anyway.

Verdict: pay Django's weight without collecting its benefits.

### Option C — Node/NestJS + React

One language everywhere, but you lose the Python data ecosystem (dbt, Snowpark, pandas) where most of the team's data logic lives, and the official dbt MCP server is Python. Rejected.

### Frontend

**React 18 + Vite + TypeScript.** Vite for dev speed; TanStack Query for server state; a component library (e.g. shadcn/ui or MUI) later; charting via ECharts or Recharts (see the `dataviz` guidance when we get there). If SEO/SSR ever matters (it won't for an internal tool), Next.js is the escape hatch — the API contract doesn't change.

## 3. Target architecture

```
┌───────────────────────────────── React (Vite) SPA ─────────────────────────────────┐
│  Chat + Artifacts │ Dashboards │ Writeback grids │ Alerts inbox │ Admin            │
└───────────────┬────────────────────────────────────────────────────────────────────┘
                │ HTTPS / SSE / WebSocket   (OIDC bearer tokens)
┌───────────────▼──────────────── FastAPI (modular monolith) ────────────────────────┐
│  auth (OIDC/SSO, RBAC/ABAC)                                                        │
│  chat        → Agent service (Claude, Anthropic SDK)                               │
│  dashboards  → semantic-layer queries (dbt metrics) + caching                      │
│  writeback   → validated mutations → Snowflake MERGE + audit log                   │
│  alerts      → rules engine + scheduler → email/push/Teams                         │
│                                                                                    │
│  MCP Host layer (long-lived sessions)                                              │
│   ├── dbt MCP server        (metadata, lineage, semantic layer, dbt commands)      │
│   └── Snowflake MCP server  (query execution, Cortex services)                     │
└───────┬──────────────────────────────┬─────────────────────────────────────────────┘
        │                              │
   Anthropic API                  Snowflake  ⇄  dbt (models, semantic layer)
   (claude-opus-4-8)
```

Key decisions:

1. **Modular monolith first, services later.** One FastAPI app, strict module boundaries (`app/modules/<name>/{router,service,models}`). Extract to services only when scale demands.
2. **Backend is the MCP host.** The server owns MCP sessions and credentials; the browser never talks to Snowflake or holds keys. Users get an audited, permission-filtered tool surface. This also means *any* agent (Claude, scheduled jobs, evaluation harnesses) reuses the same tool layer.
3. **Agent loop server-side, streamed to the client** as typed SSE events (`text_delta`, `tool_call`, `tool_result`, `done`) so the UI can render tool activity, data tables, and artifacts distinctly.
4. **Writeback is a first-class module, not a chat tool (initially).** Grid UI → optimistic validation → `MERGE` into Snowflake writeback schema → audit table. Later, expose it to the agent as a gated MCP tool with human confirmation.
5. **AuthN via OIDC (Azure AD)**, AuthZ as a policy layer mapping app roles → allowed modules, dbt marts, and Snowflake row-access policies.

## 4. What the PoC proves (this repo, today)

- FastAPI backend hosting **two MCP servers over stdio** (dbt + Snowflake), aggregating their tools with namespacing (`dbt__*`, `snowflake__*`).
- A **Claude agentic loop** (`claude-opus-4-8`, adaptive thinking, manual tool loop, streaming) that: reads dbt metadata → constructs SQL → runs it on Snowflake → returns the data plus a formatted narrative answer.
- **SSE streaming** end-to-end into a React chat UI that renders text, tool calls, and result tables.
- **Mock mode** (default): local mock dbt/Snowflake MCP servers backed by DuckDB with seeded meat-industry data, so the full flow runs with only an `ANTHROPIC_API_KEY`. Flip `MCP_MODE=real` + env vars to attach the official `dbt-mcp` and Snowflake MCP servers — the application code does not change, which is the point of MCP.

## 5. Out of scope for the PoC (next steps)

Auth/SSO, persistence of conversations, writeback UI, dashboards, alerts, artifact rendering, rate limiting, observability. Each maps onto the module layout above.
